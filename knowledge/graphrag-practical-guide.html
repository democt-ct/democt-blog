1. 传统 RAG 的痛点

1.1 Vector RAG 的三大失败模式

失败模式 1: 全局理解缺失

  • 问题: 只能检索局部文本片段，无法捕捉数据集全貌
  • 示例: 用户问"过去一年我们团队在 Agent 项目上取得了哪些进展？"
    • Vector RAG 只能找到包含"Agent"的片段
    • 无法关联时间线、团队协作、项目演进等跨文档信息

失败模式 2: 关系推理失败

  • 问题: 无法理解实体之间的复杂关系
  • 示例: "张三和李四共同参与了哪些项目？"
    • Vector RAG 可能找到提到"张三"或"李四"的片段
    • 但无法推理"共同参与"这种关系

失败模式 3: 多跳推理失效

  • 问题: 需要多步推理的查询无法处理
  • 示例: "那个在 QCon 上演讲的工程师最近发表了什么论文？"
    • 需要从 QCon → 演讲者 → 工程师 → 论文，多跳关联
    • Vector RAG 无法实现这种链式推理

2. GraphRAG 核心原理

2.1 什么是 GraphRAG？

定义: GraphRAG = Knowledge Graph + RAG

  • Knowledge Graph (知识图谱): 用图结构存储实体和关系

    • 节点 (Node): 实体（人、地点、概念、项目等）
    • 边 (Edge): 实体之间的关系（参与、属于、依赖等）

  • RAG (检索增强生成): 用知识图谱增强 LLM 的检索能力

2.3 核心技术组件

2.3.1 实体关系抽取

  • 工具: LLM (GPT-4.5, Claude Sonnet 5, DeepSeek V4)
  • 输出格式:
    {
  "entities": [
    {"id": "E1", "type": "Person", "name": "张三"},
    {"id": "E2", "type": "Project", "name": "Agent 研究"}
  ],
  "relations": [
    {"from": "E1", "to": "E2", "type": "参与", "context": "..."}
  ]
}

2.3.2 知识图谱存储

  • Neo4j (推荐): 原生图数据库，支持 Cypher 查询
  • Nebula Graph: 分布式图数据库，适合大规模数据
  • ArangoDB: 多模型数据库，支持图+文档混合存储

2.3.3 社区检测

  • 算法: Leiden 算法 (改进版 Louvain)
  • 作用: 将图谱划分为紧密相关的社区，便于分层摘要

from langchain.llms import DeepSeek

chunks = splitter.split_documents(documents)

# 2. 实体关系抽取 Prompt
extract_prompt = """
从以下文本中提取实体和关系，输出 JSON 格式：

文本：{text}

要求：
1. 识别实体类型：Person, Project, Paper, Concept, Tool
2. 识别关系类型：参与、提出、使用、依赖、改进
3. 输出格式：

  "entities": [{{"id": "E1", "type": "Person", "name": "张三"}}],
  "relations": [{{"from": "E1", "to": "E2", "type": "参与"}}]

"""

# 3. 调用 LLM 抽取
llm = DeepSeek(model="deepseek-v4")
for chunk in chunks:
    result = llm(extract_prompt.format(text=chunk.page_content))

Step 3: 写入 Neo4j

from neo4j import GraphDatabase

class Neo4jWriter:
    def __init__(self, uri, user, password):
        self.driver = GraphDatabase.driver(uri, auth=(user, password))
    
    def write_graph(self, entities, relations):
        with self.driver.session() as session:
            # 写入实体
            for entity in entities:
                session.run(
                    "MERGE (e:{type} {{id: $id}}) SET e.name = $name".format(type=entity["type"]),
                    id=entity["id"],
                    name=entity["name"]
                )
            
            # 写入关系
            for rel in relations:
                session.run(
                    """
                    MATCH (a {{id: $from_id}})
                    MATCH (b {{id: $to_id}})
                    MERGE (a)-[r:{rel_type}]->(b)
                    """.format(rel_type=rel["type"]),
                    from_id=rel["from"],
                    to_id=rel["to"]
                )

# 使用
writer = Neo4jWriter("bolt://localhost:7687", "neo4j", "password")
writer.write_graph(graph_data["entities"], graph_data["relations"])

Step 4: 查询阶段 - 自然语言转 Cypher

def natural_language_to_cypher(query: str) -> str:
    """将自然语言转换为 Cypher 查询"""
    prompt = f"""
    将以下自然语言查询转换为 Neo4j Cypher 查询：
    
    查询：{query}
    
    数据库 Schema：
    - 节点标签：Person, Project, Paper, Concept, Tool
    - 关系类型：参与、提出、使用、依赖、改进
    
    示例：
    输入："张三参与了哪些项目？"
    输出：MATCH (p:Person {{name: '张三'}})-[:参与]->(proj:Project) RETURN proj.name
    
    只输出 Cypher 语句，不要输出其他内容。
    """
    cypher = llm(prompt)
    return cypher.strip()

# 执行查询
query = "张三和李四共同参与了哪些项目？"
cypher = natural_language_to_cypher(query)

with driver.session() as session:
    result = session.run(cypher)
    for record in result:
        print(record)

Step 5: 混合检索（向量 + 图谱）

from langchain.vectorstores import Milvus
from langchain.embeddings import DeepSeekEmbeddings

# 1. 向量检索（初步筛选）
embeddings = DeepSeekEmbeddings(model="deepseek-v4-embedding")
vector_db = Milvus(embeddings, collection_name="agent_kb")
vector_results = vector_db.similarity_search(query, k=10)

# 2. 图谱检索（精确关系）
cypher_results = session.run(cypher)

# 3. 合并结果
context = {
    "vector_results": vector_results,
    "graph_results": cypher_results
}

# 4. LLM 生成答案
answer_prompt = """
基于以下上下文回答问题：

上下文：
{context}

问题：{query}

要求：
1. 综合向量检索和图谱检索的结果
2. 如果图谱中有明确的关系路径，优先使用
3. 给出可解释的答案（引用具体实体和关系）
"""
answer = llm(answer_prompt.format(context=context, query=query))

5. 高级优化技巧

5.1 增量更新策略

问题: 知识库持续更新，如何避免全量重建图谱？

解决方案: 增量抽取 + 图谱合并

def incremental_update(new_documents):
    """增量更新知识图谱"""
    # 1. 抽取新文档的实体关系
    new_graph_data = extract_entities(new_documents)
    
    # 2. 实体对齐（避免重复）
    for entity in new_graph_data["entities"]:
        # 使用名字相似度 + 类型匹配进行对齐
        existing = session.run(
            "MATCH (e:{type}) WHERE e.name CONTAINS $name RETURN e".format(type=entity["type"]),
            name=entity["name"]
        )
        if existing:
            entity["id"] = existing.single()["e"]["id"]  # 使用已有 ID
    
    # 3. 合并到已有图谱
    writer.write_graph(new_graph_data["entities"], new_graph_data["relations"])

5.2 社区摘要生成

作用: 提升全局查询的性能（如"总结过去一年的研究进展"）

def generate_community_summaries():
    """为每个社区生成摘要"""
    # 1. 使用 Leiden 算法检测社区
    communities = session.run("""
        CALL gds.leiden.stream('myGraph')
        YIELD nodeId, communityId
        RETURN gds.util.asNode(nodeId).name AS entity, communityId
    """)
    
    # 2. 为每个社区生成摘要
    for community_id, entities in communities.groupby("communityId"):
        # 提取社区内的所有实体和关系
        subgraph = extract_subgraph(community_id)
        
        # LLM 生成摘要
        summary_prompt = f"总结以下实体和关系：{subgraph}"
        summary = llm(summary_prompt)
        
        # 存储摘要
        session.run(
            "CREATE (s:Summary {community_id: $cid, content: $content})",
            cid=community_id,
            content=summary
        )

6. 常见问题与解决方案

6.1 中文分词问题

问题: LLM 抽取实体时，中文分词不准确

解决方案:

  1. 使用 jieba 分词预处理
  2. 在 Prompt 中指定实体类型（降低歧义）
  3. 后处理：合并同名实体（如"张三"和"张先生"）
import jieba.posseg as pseg

def chinese_entity_recognition(text):
    words = pseg.cut(text)
    entities = []
    for word, flag in words:
        if flag in ['nr', 'ns', 'nt']:  # 人名、地名、机构名
            entities.append({"name": word, "type": flag})
    return entities

6.2 内存爆炸问题

问题: 大规模图谱构建时，Neo4j 内存溢出

解决方案:

  1. 分批导入（每批 1000 个实体）
  2. 使用 Neo4j 的 apoc.periodic.iterate 批量写入
  3. 调整 Neo4j 堆内存配置
// 批量写入示例
CALL apoc.periodic.iterate(
  "UNWIND $entities AS entity RETURN entity",
  "MERGE (e:Entity {id: entity.id}) SET e.name = entity.name",
  {batchSize: 1000, params: {entities: $entities}}
)

6.3 图谱稀疏性问题

问题: 抽取的图谱关系太少，无法支撑多跳推理

解决方案:

  1. 使用推理模型（如 Claude Sonnet 5）增强抽取
  2. 引入外部知识库（如 Wikidata）补充关系
  3. 使用 TransE / RotatE 等知识图谱嵌入模型预测缺失关系

7. GraphRAG vs. 其他 RAG 方案对比

维度 Vector RAG GraphRAG Hybrid RAG Vectorless RAG
复杂关系推理 ❌ ✅ ✅ ⚠️
全局理解 ❌ ✅ ✅ ✅
实现复杂度 低 高 中 中
查询延迟 低 中 中 高
可解释性 低 高 中 中

选型建议:

  • 简单 FAQ / 单文档 QA: Vector RAG
  • 复杂关系查询 / 多跳推理: GraphRAG
  • 通用场景: Hybrid RAG (Vector + Graph)
  • 文档结构清晰: Vectorless RAG

8. 2026 年 GraphRAG 新趋势

8.1 Agentic GraphRAG

核心思想: 让 Agent 主动探索图谱，而非一次性检索

class GraphRAGAgent:
    def __init__(self, llm, neo4j_driver):
        self.llm = llm
        self.driver = neo4j_driver
    
    def answer(self, query):
        # Agent 迭代探索图谱
        current_node = self.identify_start_node(query)
        visited = set()
        path = []
        
        while not self.is_answer_found():
            # 1. 从当前节点扩展邻居
            neighbors = self.expand_neighbors(current_node)
            
            # 2. LLM 决策：下一步走向哪个邻居？
            decision_prompt = f"""
            当前节点：{current_node}
            已访问：{visited}
            邻居节点：{neighbors}
            
            问题：{query}
            
            选择下一个要访问的节点（或回答已找到）
            """
            decision = self.llm(decision_prompt)
            
            # 3. 更新路径
            if decision == "ANSWER_FOUND":
                return self.generate_answer(path)
            else:
                current_node = decision
                visited.add(current_node)
                path.append(current_node)

8.2 多模态 GraphRAG

扩展: 图谱不仅包含文本实体，还包含图像、表格、代码

  • 图像实体: 使用 CLIP 提取图像特征，构建图像相似度边
  • 表格实体: 使用表格理解模型（如 TAPAS）抽取表格关系
