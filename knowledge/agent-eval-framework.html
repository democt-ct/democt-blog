Agent 知识库

Agent 评估框架与基准测试

> **核心目标**: 系统化评估 Agent 的能力、可靠性、安全性，指导优化方向 > > **评估挑战**: Agent 行为的不确定性、环境交互的复杂性、长期目标的难以量化

核心目标: 系统化评估 Agent 的能力、可靠性、安全性，指导优化方向

评估挑战: Agent 行为的不确定性、环境交互的复杂性、长期目标的难以量化

1. 为什么需要评估 Agent？

1.1 Agent 评估的独特性

与传统软件测试不同，Agent 评估面临以下挑战：

挑战 描述 影响
非确定性 相同输入可能产生不同输出 难以复现 Bug
长期依赖 Agent 的决策影响未来状态 需要模拟完整流程
环境交互 Agent 依赖外部工具/API 需要 Mock 环境
主观判断 输出质量依赖人类偏好 难以自动化评估

1.2 评估的目的

  • 选型决策: 对比不同 Agent 框架/模型的能力
  • 持续优化: 量化改进效果，指导迭代方向
  • 生产监控: 实时检测 Agent 性能下降或异常行为
  • 合规审计: 证明 Agent 符合安全/伦理标准

2. 评估维度

2.1 任务完成度（Task Completion）

定义: Agent 是否成功完成指定任务。

指标:

  • 成功率（Success Rate）: 成功任务数 / 总任务数
  • 部分完成率（Partial Completion Rate）: 任务完成度 ≥ 50% 的占比
  • 平均完成步骤（Average Steps to Completion）: 完成任务所需的平均步数

评估方法:

def evaluate_success_rate(agent, tasks):
    successes = 0
    for task in tasks:
        result = agent.invoke(task)
        if verify_completion(result, task):
            successes += 1
    return successes / len(tasks)

示例:

  • 任务: "预订明天北京到上海的机票"
  • 成功: Agent 完成预订并返回确认号
  • 失败: Agent 只搜索了航班，未完成预订

2.2 效率（Efficiency）

定义: Agent 完成任务所需的资源消耗。

指标:

  • 时间成本: 任务执行时间（秒）
  • Token 消耗: LLM 调用的 Token 数
  • 工具调用次数: 外部工具/API 的调用次数
  • 成本: API 费用（美元）

评估方法:

def evaluate_efficiency(agent, task):
    start_time = time.time()
    result = agent.invoke(task)
    elapsed_time = time.time() - start_time
    
    return {
        "time": elapsed_time,
        "tokens": result["token_usage"],
        "tool_calls": len(result["tool_calls"]),
        "cost": result["cost"]
    }

优化方向:

  • 减少不必要的工具调用
  • 使用更高效的 Prompt
  • 缓存重复查询结果

2.3 鲁棒性（Robustness）

定义: Agent 在异常/对抗环境下的稳定性。

测试场景:

  • 输入扰动: 拼写错误、模糊描述、多语言混合
  • 工具故障: API 超时、返回错误、格式变化
  • 环境变化: 时区、网络延迟、并发请求
  • 对抗攻击: Prompt 注入、误导性指令

指标:

  • 故障恢复率: 遇到错误后成功恢复的比例
  • 降级成功率: 在部分工具不可用时的成功率
  • 对抗成功率: 抵御对抗攻击的成功率

评估方法:

def evaluate_robustness(agent, task, perturbation_func):
    original_result = agent.invoke(task)
    
    # 添加扰动
    perturbed_task = perturbation_func(task)
    perturbed_result = agent.invoke(perturbed_task)
    
    # 对比结果一致性
    consistency = compute_consistency(original_result, perturbed_result)
    return consistency

2.4 安全性（Safety）

定义: Agent 是否会产生有害输出或执行危险操作。

风险类型:

  • 数据泄露: 输出包含敏感信息（密码、API Key）
  • 权限滥用: 执行超出授权范围的操作
  • 恶意代码: 生成有害代码（删除文件、反弹 Shell）
  • 偏见与歧视: 输出包含性别/种族/地域歧视

指标:

  • 安全违规率: 产生不安全输出的比例
  • 权限越界率: 执行未授权操作的比例
  • 有害内容率: 输出有害内容的比例

评估方法:

def evaluate_safety(agent, task, safety_checker):
    result = agent.invoke(task)
    
    # 安全检查
    issues = safety_checker.check(result)
    
    return {
        "safe": len(issues) == 0,
        "issues": issues
    }

安全评估工具:

  • LangGuard: 实时检测 Prompt 注入
  • Rebuff: 多层防御对抗攻击
  • NeMo Guardrails: 定义安全边界

2.5 泛化能力（Generalization）

定义: Agent 在未见过的任务/领域上的表现。

测试方法:

  • 跨领域测试: 在训练/开发时未涉及的领域测试
  • 少样本测试: 只提供少量示例，测试学习能力
  • 零样本测试: 不提供示例，测试推理能力

指标:

  • 跨领域成功率: 在新领域上的成功率
  • 少样本学习曲线: 随示例数增加的性能提升
  • 泛化差距: 训练领域与测试领域的性能差异

3. 主流评估框架

3.1 LangSmith

定位: LangChain 官方评估与监控平台。

核心功能:

  • Trace 可视化: 查看 Agent 每次调用的完整链路
  • Dataset 管理: 组织测试集，版本控制
  • 自动化评估: 运行测试集，生成评估报告
  • 在线监控: 实时检测生产环境异常

使用示例:

from langsmith import Client
from langchain_openai import ChatOpenAI

# 初始化 Client
client = Client()

# 创建 Dataset
dataset = client.create_dataset(
    dataset_name="agent_evaluation_v1",
    description="Test suite for agent evaluation"
)

# 添加测试用例
client.create_examples(
    inputs=[{"task": "Book a flight to Shanghai"}],
    outputs=[{"expected": "Flight booked successfully"}],
    dataset_id=dataset.id
)

# 运行评估
llm = ChatOpenAI(model="gpt-4")

def evaluate_example(example):
    result = llm.invoke(example.inputs["task"])
    score = compute_similarity(result, example.outputs["expected"])
    return {"score": score}

run = client.run_on_dataset(
    dataset_name="agent_evaluation_v1",
    llm=llm,
    evaluation=evaluate_example
)

print(f"Average score: {run['average_score']}")

优点:

  • 与 LangChain/LangGraph 深度集成
  • 可视化界面友好
  • 支持自定义评估指标

缺点:

  • 主要支持 LangChain 生态
  • 付费功能较多

3.2 OpenAI Evals

定位: OpenAI 开源的评估框架，支持任意 LLM。

核心功能:

  • Eval 定义: 用 YAML/JSON 定义测试集和评估逻辑
  • 自动评分: 支持多种评分器（模型评分、字符串匹配、代码执行）
  • 对比实验: 并行运行多个模型，对比性能

使用示例:

{
  "eval_name": "code_generation",
  "description": "Evaluate code generation ability",
  "data_path": "test_cases.jsonl",
  "eval_type": "model_graded",
  "model": "gpt-4",
  "prompt": "Rate the code quality from 1 to 10"
}

# 运行评估
openai eval run code_generation --model gpt-4 --n 5

优点:

  • 支持任意 LLM（不限于 OpenAI）
  • 评估逻辑灵活（支持自定义评分器）
  • 开源免费

缺点:

  • 需要手动定义 Eval
  • 可视化能力较弱

3.3 DeepEval

定位: 专注于 LLM 应用的质量评估框架。

核心功能:

  • 指标体系: 内置 40+ 评估指标（准确性、相关性、有害性、偏见等）
  • 测试用例生成: 自动生成边缘案例
  • 红队测试: 模拟对抗攻击
  • CI/CD 集成: 与 GitHub Actions、Jenkins 集成

使用示例:

from deepeval import assert_test
from deepeval.test_case import LLMTestCase
from deepeval.metrics import AnswerRelevancyMetric

def test_agent_response():
    test_case = LLMTestCase(
        input="What is the capital of France?",
        actual_output="The capital of France is Paris.",
        expected_output="Paris"
    )
    
    metric = AnswerRelevancyMetric(threshold=0.7)
    assert_test(test_case, [metric])

优点:

  • 指标丰富，开箱即用
  • 支持红队测试（安全性评估）
  • 与 CI/CD 集成方便

缺点:

  • 主要面向 LLM 输出评估，对 Agent 工具调用支持有限
  • 部分高级功能需要付费

3.4 RAGAS

定位: 专注于 RAG（检索增强生成）系统的评估框架。

核心指标:

  • Faithfulness: 生成内容是否基于检索的文档
  • Answer Relevancy: 答案与问题的相关性
  • Context Precision: 检索的上下文是否包含答案
  • Context Recall: 检索的上下文是否完整

使用示例:

from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy

# 准备测试数据
test_data = {
    "question": ["What is RAG?"],
    "answer": ["RAG is Retrieval-Augmented Generation."],
    "contexts": [["RAG combines retrieval and generation..."]],
    "ground_truth": ["RAG is a technique that combines retrieval and generation."]
}

# 运行评估
results = evaluate(test_data, metrics=[faithfulness, answer_relevancy])
print(results)

优点:

  • 专为 RAG 设计，指标针对性强
  • 支持自动化评估（无需人工标注）

缺点:

  • 仅适用于 RAG 系统
  • 对非 RAG Agent 不适用

3.5 自定义评估框架

对于特定业务场景，可能需要自定义评估框架。

核心组件:

  1. Test Case 管理: 存储、版本控制、检索测试用例
  2. Executor: 执行 Agent，收集结果
  3. Scorer: 评分逻辑（规则、模型、人工）
  4. Reporter: 生成评估报告（JSON、HTML、Markdown）

示例架构:

class AgentEvaluator:
    def __init__(self, agent, test_cases):
        self.agent = agent
        self.test_cases = test_cases
    
    def run(self):
        results = []
        for case in self.test_cases:
            result = self.agent.invoke(case["input"])
            score = self.score(result, case["expected"])
            results.append({
                "case": case,
                "result": result,
                "score": score
            })
        return self.aggregate(results)
    
    def score(self, result, expected):
        # 自定义评分逻辑
        return compute_similarity(result, expected)
    
    def aggregate(self, results):
        return {
            "average_score": sum(r["score"] for r in results) / len(results),
            "success_rate": sum(1 for r in results if r["score"] >= 0.8) / len(results)
        }

4. 基准测试（Benchmark）

4.1 SWE-bench

定位: 评估 Agent 解决真实 GitHub Issue 的能力。

数据集:

  • 2,294 个来自 12 个 Python 项目的 Issue
  • 每个 Issue 包含问题描述、代码库快照、测试用例

评估指标:

  • 解决率: 正确修复 Issue 的比例
  • 测试用例通过率: 生成的代码通过所有测试的比例

SOTA 成绩（2026-07）:

  • Claude Fable 5: 95% (SWE-bench Verified)
  • OpenDevin (CodeActAgent): 60%
  • AutoCodeRover: 56%

使用方法:

# 安装 SWE-bench
pip install swebench

# 运行评估
python -m swebench.run \
    --agent my_agent.py \
    --dataset swe_bench_lite \
    --output results.json

4.2 AgentBench

定位: 多领域 Agent 评估基准。

涵盖领域:

  • 操作系统: 命令行任务（如文件操作、进程管理）
  • 数据库: SQL 查询生成与执行
  • 知识图谱: 推理与问答
  • 卡牌游戏: 策略决策（如 UNO）
  • 网页导航: 模拟用户浏览网页

评估指标:

  • 任务完成率: 各领域的平均完成率
  • 平均步数: 完成任务所需的平均步数
  • 容错率: 遇到错误后的恢复能力

使用示例:

from agentbench import AgentBench

bench = AgentBench()

# 运行单个领域评估
results = bench.evaluate(
    agent=my_agent,
    domain="operating_system",
    num_samples=100
)

print(f"Success rate: {results['success_rate']}")

4.3 ToolBench

定位: 评估 Agent 使用工具的能力。

数据集:

  • 包含 5,000+ 个真实 API（来自 RapidAPI）
  • 每个 API 包含文档、示例请求/响应

评估维度:

  • API 选择: 能否选择正确的 API
  • 参数填充: 能否正确填充 API 参数
  • 错误处理: 遇到 API 错误时能否重试/切换

指标:

  • 成功率: 正确调用 API 并完成任务的比例
  • 效率: 完成任务所需的 API 调用次数
  • 鲁棒性: API 返回错误时的恢复率

5. 评估最佳实践

5.1 构建测试集

原则:

  • 多样性: 覆盖不同难度、领域、场景
  • 代表性: 反映真实用户需求
  • 可扩展: 易于添加新测试用例

测试用例结构:

{

5.2 自动化评估流程

graph LR
    A[构建测试集] --> B[运行 Agent]
    B --> C[收集结果]
    C --> D[自动评分]
    D --> E[生成报告]
    E --> F{通过阈值?}

集成 CI/CD:

# .github/workflows/evaluate.yml
name: Agent Evaluation

on: [push, pull_request]

jobs:
  evaluate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Run evaluation
        run: |
          python evaluate_agent.py --test-suite regression_tests.json
      - name: Check threshold
        run: |
          if [ $(cat results.json | jq '.success_rate') -lt 0.8 ]; then
            echo "Success rate below threshold!"
            exit 1
          fi

5.3 人工评估

何时需要人工评估:

  • 自动评分器无法判断输出质量（如创意写作、设计建议）
  • 需要评估主观维度（如友好性、同理心）
  • 自动评分器本身需要验证

人工评估流程:

  1. 抽样: 从自动评估结果中抽样（如 10%）
  2. 标注: 评估人员根据评分标准打分
  3. 对齐: 对比自动评分与人工评分，计算一致性（Cohen's κ）
  4. 校准: 调整自动评分器参数，提升对齐度

评分标准示例:

维度 1 分 2 分 3 分 4 分 5 分
相关性 完全无关 部分相关 基本相关 高度相关 完全相关
完整性 完全缺失 部分缺失 基本完整 高度完整 完全完整
可读性 无法理解 难以理解 基本可读 易于理解 非常流畅

5.4 A/B 测试

场景: 对比两个 Agent 版本（如 v1.0 vs v2.0）。

设计:

  1. 流量分配: 将用户请求随机分配到两个版本（如 50:50）
  2. 指标收集: 收集成功率、用户满意度、响应时间等指标
  3. 统计分析: 使用 T 检验或卡方检验判断差异是否显著

示例:

import random

def ab_test(user_request):
    # 50% 流量到 v1.0，50% 到 v2.0
    if random.random() < 0.5:
        result = agent_v1.invoke(user_request)
        version = "v1.0"
    else:
        result = agent_v2.invoke(user_request)
        version = "v2.0"
    
    # 记录结果
    log_result(user_request, result, version)
    return result

5.5 持续监控

生产环境监控指标:

  • 成功率: 实时成功率，设置告警阈值（如 < 80%）
  • 响应时间: P50、P95、P99 响应时间
  • Token 消耗: 每日/每月 Token 消耗，预测成本
  • 用户反馈: 用户点赞/点踩率、满意度评分

监控工具:

  • LangSmith: 实时 Trace 监控
  • Prometheus + Grafana: 自定义指标监控
  • Sentry: 异常捕获与告警

告警规则示例:

# prometheus/alerts.yml
groups:
  - name: agent_alerts
    rules:
      - alert: LowSuccessRate
        expr: success_rate < 0.8
        for: 5m
        annotations:
          summary: "Agent success rate below 80%"
      
      - alert: HighLatency
        expr: p95_response_time > 10s
        for: 5m
        annotations:
          summary: "Agent P95 latency above 10 seconds"

   - **改进建议**: 增加澄清逻辑，询问用户预算
