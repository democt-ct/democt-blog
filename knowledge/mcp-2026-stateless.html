MCP

MCP 2026 无状态架构迁移指南

**Model Context Protocol (MCP)** 是由 Anthropic 于 2024 年 11 月推出的开放标准协议，旨在为大型语言模型（LLM）与外部工具、数据源之间提供标准化的通信方式。MCP 采用客户端-主机-服务器（Client-Host-Server）三层架构，基于 JSON-RPC 2.

1. 概述

1.1 什么是 MCP

Model Context Protocol (MCP) 是由 Anthropic 于 2024 年 11 月推出的开放标准协议，旨在为大型语言模型（LLM）与外部工具、数据源之间提供标准化的通信方式。MCP 采用客户端-主机-服务器（Client-Host-Server）三层架构，基于 JSON-RPC 2.0 进行消息编码，通过 Tools（工具）、Resources（资源）、Prompts（提示模板）三大原语暴露能力。MCP 常被类比为「AI 世界的 USB-C 接口」。

1.2 MCP 协议的版本演进

版本 发布日期 关键变更
2024-11-05 2024-11 初始版本，定义基础协议、stdio/SSE 传输、三大原语
2025-03-26 2025-03-26 引入 OAuth 2.1 授权框架；新增 Streamable HTTP 传输层
2025-06-18 2025-06-18 正式发布 Authorization 规范章节；SSE 逐步边缘化
2025-11-25 2025-11-25 增强授权流程、增量 Scope 协商、OAuth Client ID Metadata 等
2026-07-28 (RC) 2026-05-21 (RC发布) 无状态核心架构、Extensions 框架、Tasks、MCP Apps、正式弃用策略

注意：2026-07-28 规范在 2026-05-21 发布了 Release Candidate，这是 MCP 自诞生以来最大的一次协议修订。本章基于此 RC 版本及社区讨论撰写，部分细节可能随正式版发布调整。

1.3 2026 规范的核心变更

  1. 无状态核心（Stateless Core）：去除服务端会话管理，每个请求独立鉴权，使 MCP Server 可以在普通 HTTP 基础设施上水平扩展。
  2. OAuth 2.1 全面接入：PKCE 强制、隐式流废除、动态客户端注册、Resource Indicators (RFC 8707)。
  3. Extensions 框架：引入 MCP Apps（服务端渲染 UI）和 Tasks（长时间运行工作）等扩展能力。
  4. 正式弃用策略（Formal Deprecation Policy）：协议演进不影响已有实现。

2. 核心变更详解

2.1 去除会话管理：Why & How

2.1.1 旧版架构的问题

在 2024-11-05 到 2025-06-18 期间，基于 SSE（Server-Sent Events）的 MCP Server 使用 Mcp-Session-Id HTTP Header 来维护客户端-服务端之间的会话状态：

旧的 SSE 模式：
┌──────────┐    GET /sse (建立 SSE 长连接)    ┌──────────┐
│  Client  │ ←─────────────────────────────→ │  Server  │
│          │    POST /message?sessionId=xxx   │ (有状态) │
└──────────┘    (携带 Session ID 发消息)       └──────────┘

固有缺陷：

  • 无法水平扩展：会话状态绑定在单个 Server 进程/实例上，无法在负载均衡器后方运行多个副本。
  • 连接状态耦合：客户端与 Server 之间存在长连接依赖，任何一端断开都导致会话丢失。
  • 资源消耗：每个活跃客户端需要 Server 维护一个常驻连接和内存状态。
  • 不兼容 Serverless：无法部署在 AWS Lambda、Cloudflare Workers 等无状态计算平台上。
  • 安全模型不清晰：Session ID 本质上是一种不透明的 Bearer Token，但其管理机制不规范。

2.1.2 新架构的设计

2026-07-28 RC 规范将 MCP 核心定义为完全无状态：

新的无状态模式：
┌──────────┐    POST /mcp (Authorization: Bearer <token>)   ┌──────────┐
│  Client  │ ←────────────────────────────────────────────→ │  Server  │
│          │    每个请求独立鉴权，无服务端 Session            │ (无状态) │
└──────────┘                                                └──────────┘
                                     ↕
                            ┌──────────────────┐
                            │   负载均衡器      │
                            └──┬───────┬───────┘
                               ↓       ↓
                          ┌────────┐ ┌────────┐
                          │Server A│ │Server B│
                          └────────┘ └────────┘

关键特性：

  • 每个 HTTP 请求独立：Authorization: Bearer <access-token> 必须出现在每一个请求中。
  • 无服务端状态：Server 不保存客户端会话上下文，所有必要状态由 Client 在请求中携带。
  • 可水平扩展：任何无状态的 MCP Server 实例可以处理任何请求，适合 K8s、Serverless 部署。
  • HTTP-only 传输：Streamable HTTP 成为远程通信的标准传输方式。

2.1.3 对 Server 实现的影响

需要移除旧版 Server 中的以下实现：

移除项：
  ✗ Mcp-Session-Id header 的生成和管理
  ✗ 会话存储（内存 Map / Redis / 数据库）
  ✗ SSE 长连接维护逻辑
  ✗ 会话超时与清理定时任务
  ✗ 客户端重连时的会话恢复逻辑

2.2 OAuth 2.1 全面接入

MCP 的授权机制基于 OAuth 2.1 IETF 草案 (draft-ietf-oauth-v2-1-13)，并补充了以下 RFC 标准：

标准 用途
RFC 8414 Authorization Server Metadata — 授权服务器元数据发现
RFC 7591 Dynamic Client Registration — 动态客户端注册
RFC 9728 Protected Resource Metadata — 资源服务器元数据
RFC 8707 Resource Indicators — Token 与资源绑定

2.2.1 PKCE 强制执行

OAuth 2.1 中，PKCE（Proof Key for Code Exchange） 对所有使用 Authorization Code Flow 的客户端都是强制性的，包括公开客户端（Public Clients，如 SPA/CLI）。

# PKCE 参数生成 (Python)
import hashlib, base64, os

def generate_code_verifier():
    """生成 code_verifier: 43-128 字符的随机字符串"""
    return base64.urlsafe_b64encode(os.urandom(32)).rstrip(b'=').decode()

def generate_code_challenge(verifier: str) -> str:
    """生成 code_challenge: SHA256(code_verifier) 的 Base64URL 编码"""
    digest = hashlib.sha256(verifier.encode()).digest()
    return base64.urlsafe_b64encode(digest).rstrip(b'=').decode()

在授权请求和 Token 请求中：

# 授权请求 (浏览器)
GET /authorize?
  response_type=code&
  client_id=xxx&
  code_challenge=<challenge>&
  code_challenge_method=S256&
  resource=https%3A%2F%2Fmcp.example.com

# Token 请求 (服务端)
POST /token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code&
code=<auth_code>&
code_verifier=<verifier>&
resource=https://mcp.example.com

2.2.2 隐式流 (Implicit Grant) 废除

OAuth 2.1 正式废除了 Implicit Grant Flow。所有 MCP 客户端 必须 使用 Authorization Code Flow + PKCE，不再支持直接从授权端点获取 Access Token。

2.2.3 DPoP (Demonstrating Proof-of-Possession) 支持

⚠️ 待验证：截至 2026-07，DPoP (RFC 9449) 在 MCP 规范中的支持状态尚未在公开 spec 草案中明确。社区讨论表明 DPoP 可能作为 Extensions 的一部分在未来引入，用于增强 Token 持有者证明。以下为前瞻性内容。

DPoP 的核心价值：将 Access Token 绑定到特定的非对称密钥对，即使 Token 泄露，攻击者也无法使用（因为没有私钥）。在 MCP 场景中，这对工具调用特别重要——一个泄露的 Bearer Token 可能允许攻击者通过 MCP Server 执行危险操作（文件删除、数据库修改等）。

2.3.2 Token 验证流程

MCP Server 作为 OAuth 2.1 Resource Server，在每个请求中：

  1. 提取 Authorization: Bearer <token> 头
  2. 验证 JWT 签名（使用 Authorization Server 的公钥 / JWKS）
  3. 验证 aud (audience) — Token 是否颁发给本 Server
  4. 验证 exp (expiration) — Token 是否过期
  5. 验证 scope — Token 是否包含所需权限
  6. 失败则返回 HTTP 401 或 HTTP 403
# MCP Server Token 验证中间件 (Python/FastAPI 示例)
from fastapi import Request, HTTPException
from jose import jwt, JWTError
import httpx

class MCPAuthMiddleware:
    def __init__(self, jwks_url: str, expected_audience: str):
        self.jwks_url = jwks_url
        self.expected_audience = expected_audience

    async def __call__(self, request: Request):
        auth_header = request.headers.get("Authorization")
        if not auth_header or not auth_header.startswith("Bearer "):
            raise HTTPException(status_code=401,
                headers={"WWW-Authenticate": f'Bearer resource_metadata="/.well-known/oauth-protected-resource"'})

        token = auth_header.split(" ")[1]

        # 获取 JWKS
        async with httpx.AsyncClient() as client:
            resp = await client.get(self.jwks_url)
            jwks = resp.json()

        try:
            payload = jwt.decode(
                token,
                jwks,
                audience=self.expected_audience,
                algorithms=["RS256", "ES256"],
                options={"verify_exp": True}
            )
        except JWTError as e:
            raise HTTPException(status_code=401, detail=f"Invalid token: {e}")

        request.state.user_id = payload.get("sub")
        request.state.scopes = payload.get("scope", "").split()

3.2 新版 MCP Server 实现范例

3.2.1 Node.js (TypeScript) 实现

// mcp-server-stateless.ts
// 使用 @modelcontextprotocol/sdk v2.x (待验证确切 API)

import { McpServer, StreamableHTTPServerTransport } from "@modelcontextprotocol/sdk/server/index.js";
import * as jose from "jose";

interface TokenClaims {
  sub: string;
  aud: string;
  exp: number;
  scope: string;
}

class StatelessMcpServer {
  private jwks: jose.JSONWebKeySet;
  private audience: string;

  constructor(jwksUrl: string, audience: string) {
    this.audience = audience;
    this.loadJwks(jwksUrl);
  }

  private async loadJwks(url: string) {
    const response = await fetch(url);
    this.jwks = await response.json();
  }

  // Token 验证中间件
  async verifyToken(authHeader: string): Promise<TokenClaims> {
    if (!authHeader?.startsWith("Bearer ")) {
      throw { status: 401, message: "Missing or invalid Authorization header" };
    }

    const token = authHeader.slice(7);
    try {
      const { payload } = await jose.jwtVerify(token, jose.createLocalJWKSet(this.jwks), {
        audience: this.audience,
        requiredClaims: ["sub", "exp"],
      });
      return payload as unknown as TokenClaims;
    } catch (err) {
      throw { status: 401, message: `Token validation failed: ${err}` };
    }
  }

  // 创建 Streamable HTTP Server
  async start(port: number = 3000) {
    const server = new McpServer({
      name: "stateless-mcp-demo",
      version: "1.0.0",
    });

    // 注册工具 — 与旧版完全相同的 API
    server.tool("calculator", "执行数学计算", {
      expression: { type: "string", description: "数学表达式" }
    }, async (args) => {
      try {
        const result = eval(args.expression); // 生产环境使用安全沙箱
        return {
          content: [{ type: "text", text: String(result) }]
        };
      } catch (e) {
        return {
          content: [{ type: "text", text: `Error: ${e}` }],
          isError: true
        };
      }
    });

    const transport = new StreamableHTTPServerTransport({
      endpoint: "/mcp",
      authenticate: async (authHeader: string) => {
        const claims = await this.verifyToken(authHeader);
        return { userId: claims.sub, scopes: claims.scope?.split(" ") ?? [] };
      }
    });

    await server.connect(transport);
    console.log(`MCP Server running on http://localhost:${port}/mcp`);
  }
}

// 启动
const srv = new StatelessMcpServer(
  "https://auth.example.com/.well-known/jwks.json",
  "https://mcp.example.com"
);
srv.start(3000);

3.2.2 Python 实现

# mcp_server_stateless.py
# 使用 mcp SDK v2.x (待验证确切 API)

from mcp.server import MCPServer, StreamableHTTPTransport
from mcp.server.auth import TokenValidator
import httpx
from jose import jwt
from jose.exceptions import JWTError

class StatelessTokenValidator(TokenValidator):
    def __init__(self, jwks_url: str, audience: str):
        self.jwks_url = jwks_url
        self.audience = audience
        self._jwks = None

    async def _get_jwks(self):
        if self._jwks is None:
            async with httpx.AsyncClient() as client:
                resp = await client.get(self.jwks_url)
                self._jwks = resp.json()
        return self._jwks

    async def validate(self, token: str) -> dict:
        jwks = await self._get_jwks()
        try:
            payload = jwt.decode(
                token, jwks,
                audience=self.audience,
                algorithms=["RS256", "ES256"],
                options={"verify_exp": True, "verify_aud": True}
            )
            return {
                "user_id": payload.get("sub"),
                "scopes": payload.get("scope", "").split(),
            }
        except JWTError as e:
            raise PermissionError(f"Token validation failed: {e}")

# 创建 Server
server = MCPServer(
    name="stateless-demo",
    version="1.0.0",
    transport=StreamableHTTPTransport(endpoint="/mcp"),
    token_validator=StatelessTokenValidator(
        jwks_url="https://auth.example.com/.well-known/jwks.json",
        audience="https://mcp.example.com"
    )
)

@server.tool("get_weather")
async def get_weather(city: str) -> str:
    """获取城市天气"""
    # 业务逻辑不变
    return f"The weather in {city} is sunny, 22°C."

if __name__ == "__main__":
    server.run(host="0.0.0.0", port=8080)

MCP Client 端同样需要适配：

3.3.1 核心变更

3.3.2 Client 伪代码

# MCP Client 适配示例 (Python)
import httpx

class StatelessMCPClient:
    def __init__(self, server_url: str):
        self.server_url = server_url
        self.access_token: str | None = None
        self.refresh_token: str | None = None

    async def _ensure_auth(self):
        """确保有有效的 access token"""
        if self.access_token:
            # 测试 token 是否仍然有效
            resp = await self._make_request("ping")
            if resp.status_code != 401:
                return

        # 需要重新授权 — 启动 OAuth 2.1 流程
        await self._perform_oauth_flow()

    async def _perform_oauth_flow(self):
        """完整的 OAuth 2.1 Authorization Code + PKCE 流程"""
        # 1. 从 MCP Server 获取 Authorization Server 信息
        resp = await httpx.AsyncClient().get(
            f"{self.server_url}/.well-known/oauth-protected-resource"
        )
        resource_meta = resp.json()
        as_url = resource_meta["authorization_servers"][0]

        # 2. 获取 AS 元数据
        resp = await httpx.AsyncClient().get(
            f"{as_url}/.well-known/oauth-authorization-server"
        )
        as_meta = resp.json()

        # 3. 动态注册 (RFC 7591)
        resp = await httpx.AsyncClient().post(
            as_meta["registration_endpoint"],
            json={"client_name": "MCP Client", "grant_types": ["authorization_code"]}
        )
        client_reg = resp.json()

        # 4. PKCE + Authorization Code 流程...
        # (浏览器跳转, 用户授权, 获取 code, 兑换 token)
        # 代码省略，参见 2.2.1 节

        self.access_token = "..."
        self.refresh_token = "..."

    async def _make_request(self, method: str, params: dict | None = None):
        headers = {}
        if self.access_token:
            headers["Authorization"] = f"Bearer {self.access_token}"
        async with httpx.AsyncClient() as client:
            return await client.post(
                self.server_url,
                json={"jsonrpc": "2.0", "id": 1, "method": method, "params": params or {}},
                headers=headers
            )

    async def call_tool(self, tool_name: str, args: dict):
        await self._ensure_auth()
        return await self._make_request("tools/call", {"name": tool_name, "arguments": args})

    async def list_tools(self):
        await self._ensure_auth()
        return await self._make_request("tools/list")

4. 兼容性处理

4.1 过渡期策略

MCP 社区明确声明了正式的弃用策略，意味着旧版传输（SSE）不会立即移除，而是有一个逐步淘汰的过程。在此过渡期内，推荐以下策略：

推荐的兼容性架构：

                    ┌─────────────────────────┐
                    │     MCP Server (v2)      │
                    │                         │
                    │  /mcp (Streamable HTTP)  │
                    │  ← OAuth 2.1 鉴权        │
                    │                         │
                    │  /mcp-legacy (SSE)       │
                    │  ← Session ID 鉴权       │
                    │  ← Deprecation Warning   │
                    └─────────────────────────┘
  • 新 Client → 直接使用 /mcp 端点 (Streamable HTTP + OAuth 2.1)
  • 旧 Client → 仍可使用 /mcp-legacy 端点 (SSE + Session ID)，但 Server 应返回 Deprecation warning header

4.3 Client 端降级策略

// Client 自动检测并适配
async function connectToMcpServer(serverUrl: string): Promise<McpClient> {
  try {
    // 先尝试新版 Streamable HTTP + OAuth
    const client = new StatelessMCPClient(serverUrl);
    await client.ping();
    return client;
  } catch (err) {
    if (err.status === 401) {
      // 启动 OAuth 2.1 流程
      await client.authorize();
      return client;
    }
    console.warn("新版连接失败，降级到 SSE 模式");
    // 降级到 SSE
    return new LegacySSEMCPClient(serverUrl);
  }
}

5. 安全最佳实践

5.1 Token 安全

  1. 短生命周期 Access Token：建议 5-15 分钟，使用 Refresh Token 轮换。
  2. Token 绑定（Audience Restriction）：通过 RFC 8707 Resource Indicators 确保 Token 仅用于特定 MCP Server。
  3. 最小权限 Scope：每个 Token 仅包含该 Client 实际需要的 scope（如 mcp:tools:read 而非 mcp:*）。
  4. Token 不可出现在 URL 中：MCP 规范明确禁止 access_token 作为 query parameter。
  5. Refresh Token Rotation：每次使用 Refresh Token 时颁发新的 Refresh Token，旧的立即失效。

5.2 传输层安全

  1. 强制 HTTPS：生产环境的所有 MCP 通信必须使用 TLS。
  2. Certificate Pinning（可选）：移动/M2M 场景建议固定证书指纹。
  3. HSTS Header：Server 应设置 Strict-Transport-Security。

5.3 Server 端防护

  1. Rate Limiting：基于 Client ID 或 IP 的请求频率限制。
  2. Request Validation：严格的 JSON Schema 验证，防止注入攻击。
  3. Tool 权限检查：不仅在 Token 层面校验 scope，还应在每个 tool 调用时二次验证权限。
  4. Audit Logging：记录所有 tool 调用（who、what、when、result）。
# Tool 级别权限检查示例
@server.tool("delete_file", scopes=["mcp:admin"])
async def delete_file(path: str, user: User):
    if "mcp:admin" not in user.scopes:
        raise PermissionError("Insufficient permissions to delete files")
    logger.info(f"User {user.id} deleted file: {path}")
    os.remove(path)
    return "File deleted"

总结

MCP 2026 无状态架构迁移的核心要点：

  1. SSE → Streamable HTTP：去除长连接依赖和 Mcp-Session-Id，每个请求独立。
  2. 自研认证 → OAuth 2.1：PKCE 强制，隐式流废除，Bearer Token 鉴权。
  3. 有状态 → 无状态：Server 可以水平扩展，适配 K8s/Serverless。
  4. 渐进式迁移：双模式兼容期间可以同时支持新旧客户端。
  5. 安全优先：Token 绑定 Audience、最小权限 Scope、Refresh Token Rotation 是最低配置。
