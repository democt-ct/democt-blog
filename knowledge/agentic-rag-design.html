RAG

Agentic RAG 架构设计

标准 RAG（Retrieval-Augmented Generation）流水线的核心流程是： graph LR A[用户提问] --> B[Query Embedding]

1. 概述

1.1 标准 RAG 的痛点

标准 RAG（Retrieval-Augmented Generation）流水线的核心流程是：

graph LR
    A[用户提问] --> B[Query Embedding]
    B --> C[向量检索 Top-K]
    C --> D[拼接 Context]
    D --> E[LLM 生成答案]

这个流程在实际生产中存在以下根本性缺陷：

痛点 表现 根因
检索漂移（Retrieval Drift） 用户问"怎么部署"，检索到一堆"安装教程"而不是"生产部署方案" query-to-embedding 的语义损失
一次性硬挤答案 检索结果完全不对口，模型仍然强行编造 没有反馈回路判断检索质量
缺乏自我纠错 生成后无法验证事实是否与检索文档一致 生成与检索解耦，无交叉验证
多跳推理断裂 "A公司的CEO是谁，他去年写了什么书？"——这种需要跨文档串联的查询无法一步完成 单轮检索无法处理复杂推理链
知识冲突无感知 检索到相互矛盾的文档时，模型随机选择一边 无冲突检测和消解机制

1.2 Agentic RAG 的核心思路

Agentic RAG 将 Agent（智能体） 的决策能力注入到 RAG 流程中，核心哲学是：

不急着从检索结果硬挤答案，而是先让 Agent 判断检索结果是否有用；没用就重写查询再检索，直到拿到高质量上下文。

Agentic RAG 赋予 RAG 的三个关键能力：

  1. 感知（Perception）：能判断检索质量，知道"这次检索不行"
  2. 决策（Decision）：能选择工具（切换检索源、改写 query、拆分子问题）
  3. 行动（Action）：能执行多轮检索、调用外部 API、验证事实

3. 核心组件详解

3.1 Query Rewriter（查询重写器）

职责：将用户的自然语言问题改写为更利于检索的查询表达式。

核心策略：

  • HyDE（Hypothetical Document Embeddings）：先用 LLM 生成一个假设性的答案文档，再用这个文档的 embedding 去检索，而不是直接用原始 query 的 embedding
  • Multi-Query Rewriting：将一个复杂问题拆解为多个视角的子查询，从不同角度覆盖语义空间
  • Step-back Prompting：把具体问题抽象为更上层的概念性问题，先检索背景知识

Prompt 模板示例：

你是一个查询重写专家。请将用户的原始问题改写为更适合向量检索的查询。

原始问题：{user_query}

要求：
1. 去除口语化表达，保留核心技术术语
2. 如果是多跳问题，拆分为多个子查询
3. 考虑同义词和术语的不同表达方式

输出格式（JSON）：
{
  "rewritten_queries": ["查询1", "查询2", ...],
  "reasoning": "改写理由"
}

3.2 Retrieval Evaluator（检索评估器）

职责：判断检索结果是否与用户意图相关，决定是否需要重新检索。

评估维度：

维度 说明 判断方式
Relevance（相关性） 文档是否与问题语义相关 LLM-as-Judge 打分（1-5）
Completeness（完整性） 检索结果是否覆盖了问题的所有信息需求 信息覆盖度检查
Contradiction（矛盾性） 多个文档之间是否存在矛盾 交叉验证与冲突检测
Freshness（时效性） 文档的时间是否满足需求 时间戳过滤

评估阈值设计：

  • threshold_high = 0.8：直接进入生成阶段
  • threshold_mid = 0.5：对低分文档进行二次筛选后使用
  • threshold_low = 0.3：触发 Query Rewrite + 重新检索

3.3 Multi-round Retriever（多轮检索器）

职责：管理多轮检索的生命周期，组合多种检索策略。

检索策略栈：

  1. Dense Retrieval：基于向量相似度的稠密检索（语义匹配）
  2. Sparse Retrieval：基于 BM25 等算法的稀疏检索（关键词匹配）
  3. Hybrid Search：Dense + Sparse 加权融合，取长补短
  4. Re-ranking：用 Cross-encoder 对粗排结果精排

Hybrid Search 加权策略：

Score_final = α × Score_dense + (1 - α) × Score_sparse

其中 α 通常取值 0.5-0.7，根据场景可调：偏语义理解加大 α，偏关键词精确匹配减小 α。

3.4 Answer Synthesizer（答案合成器）

职责：基于高质量检索结果生成最终答案，并进行事实校验。

关键能力：

  • Citation（引用标注）：回答中标注信息来源，可追溯
  • Uncertainty Expression（不确定性表达）：信息不足时明确表示不确定，而不是编造
  • Conflict Resolution（冲突消解）：面对矛盾信息时，呈现多方观点并给出可信度判断

4. 设计模式

4.1 ReAct 循环

ReAct（Reasoning + Acting）是最基础的 Agentic RAG 模式。Agent 在每轮循环中交替进行思考（Thought）→ 行动（Action）→ 观察（Observation）。

Round 1:
  Thought: 用户想知道K8s生产部署的最佳实践，我需要搜索相关文档
  Action: search("Kubernetes production deployment best practices")
  Observation: 返回5个文档，其中3个是关于基础概念的，只有2个涉及生产部署

Round 2:
  Thought: 检索结果不够精准，需要更具体的查询
  Action: search("Kubernetes 生产环境 高可用 部署 资源规划")
  Observation: 返回4个高度相关的生产部署文档

Round 3:
  Thought: 信息充分，可以合成答案
  Action: generate_answer(documents, user_query)

4.2 Self-RAG

Self-RAG 在生成的每一步都引入自我反思（Self-Reflection），通过特殊的 reflection token 控制检索和生成流程。

Reflection Token 类型：

  • [Retrieve]：判断是否需要检索
  • [IsRel]：判断检索到的文档是否相关
  • [IsSup]：判断生成内容是否被文档支撑
  • [IsUse]：判断生成内容是否有用

决策流程：

[Retrieve=Yes] → 执行检索 → [IsRel=Yes] → 生成段落 → [IsSup=Yes, IsUse=Yes] → 输出
                  ↓                        ↓
              [IsRel=No]              [IsSup=No]
                  ↓                        ↓
              重新检索                重新生成/修正

4.3 CRAG（Corrective RAG）

CRAG 引入了一个关键的检索评估器（Retrieval Evaluator），在检索后、生成前对文档质量进行判断，将检索结果分为三类：

  • Correct（正确）：文档高度相关，直接用于生成
  • Incorrect（不正确）：文档不相关或误导，丢弃
  • Ambiguous（模糊）：部分相关，需要进一步验证或补充

CRAG 的关键改进是 Knowledge Refinement：对 Ambiguous 类文档，不是简单丢弃，而是尝试从中提取有用片段，或通过 Web Search 补充额外知识。这种"纠错式"检索使系统对噪声数据的鲁棒性显著提升。

4.4 Self-Reflective RAG

在标准 Agentic RAG 的基础上，Self-Reflective RAG 增加了答案级别的自我反思：

  1. 生成后验证：答案生成后，Agent 主动审查答案的事实准确性
  2. Gap Analysis：识别答案中缺失的信息，触发补充检索
  3. Multi-perspective Review：从不同角度审视答案的完整性

对比总结：

模式 检索轮次 评估点 是否修正 核心机制
Standard RAG 1 无 否 单次检索+直接生成
ReAct 多轮 Thought 阶段 检索方向修正 Think-Act-Observe 循环
Self-RAG 按需 段落级 生成内容修正 Reflection Token 引导
CRAG 1-2轮 检索后 检索结果修正 Evaluator + Knowledge Refinement
Self-Reflective RAG 多轮 检索后+生成后 全链路修正 多阶段评估+补充检索

5. LangGraph 实现概要

5.1 为什么选择 LangGraph

LangGraph 是 LangChain 生态中的状态图框架，特别适合构建 Agentic RAG 这种有环图（Cyclic Graph）场景：

  • 原生支持循环：ReAct 等模式天然需要循环，LangGraph 的 graph-based 模型天然支持
  • 状态管理：Agent 在多轮检索中需要维护状态（重试次数、已检索文档、评估历史），LangGraph 的 StateGraph 提供了清晰的抽象
  • checkpointing：支持状态持久化，便于断点续传和调试

5.2 核心实现框架

from typing import TypedDict, List, Annotated
from langgraph.graph import StateGraph, END
import operator

# --- 状态定义 ---
class AgenticRAGState(TypedDict):
    # 用户输入
    user_query: str
    # 改写后的查询列表
    rewritten_queries: List[str]
    # 检索到的文档
    retrieved_docs: Annotated[List[str], operator.add]
    # 检索评估结果
    eval_scores: List[float]
    # 检索轮次计数
    retrieval_round: int
    # 最大检索轮次
    max_rounds: int
    # 最终答案
    final_answer: str

# --- 节点定义 ---
def query_rewriter(state: AgenticRAGState) -> AgenticRAGState:
    """查询重写节点：将用户原始问题改写为更适合检索的查询"""
    prompt = f"""将以下问题改写为更适合检索的查询，如果是复杂问题请拆分为2-3个子查询。
    问题：{state['user_query']}
    以JSON格式返回：{{"queries": ["查询1", "查询2"]}}"""
    
    response = llm.invoke(prompt)
    queries = parse_queries(response)  # 解析JSON
    return {
        "rewritten_queries": queries,
        "retrieval_round": state.get("retrieval_round", 0) + 1
    }

def retriever(state: AgenticRAGState) -> AgenticRAGState:
    """多轮检索节点：执行向量检索 + BM25 混合检索"""
    all_docs = []
    for query in state["rewritten_queries"]:
        dense_results = vector_store.similarity_search(query, k=5)
        sparse_results = bm25_retriever.search(query, k=5)
        merged = reciprocal_rank_fusion(dense_results, sparse_results)
        all_docs.extend(merged)
    
    # 去重 + Rerank
    unique_docs = deduplicate(all_docs)
    reranked = cross_encoder.rerank(query=state["user_query"], documents=unique_docs)
    return {"retrieved_docs": reranked[:10]}

def evaluator(state: AgenticRAGState) -> AgenticRAGState:
    """检索评估节点：判断检索质量"""
    prompt = f"""评估以下检索文档与用户问题的相关性（1-5分）。
    用户问题：{state['user_query']}
    检索文档：{state['retrieved_docs']}
    
    请从以下维度评估：
    1. 相关性：文档是否与问题直接相关
    2. 完整性：文档是否覆盖了所有需要的信息
    3. 一致性：文档之间是否有矛盾
    
    以JSON返回：{{"relevance": 1-5, "completeness": 1-5, "consistency": 1-5, "overall": 0.0-1.0}}"""
    
    response = llm.invoke(prompt)
    scores = parse_scores(response)
    return {"eval_scores": [scores["overall"]]}

def answer_synthesizer(state: AgenticRAGState) -> AgenticRAGState:
    """答案合成节点：基于检索结果生成最终答案"""
    prompt = f"""基于以下检索文档回答用户问题。要求：
    1. 引用具体来源（标注文档编号）
    2. 信息不足时明确说出，不要编造
    3. 存在矛盾时呈现多方观点
    
    用户问题：{state['user_query']}
    检索文档：{state['retrieved_docs']}"""
    
    response = llm.invoke(prompt)
    return {"final_answer": response}

# --- 路由逻辑 ---
def should_retry(state: AgenticRAGState) -> str:
    """判断是否需要重新检索"""
    MAX_ROUNDS = 3
    QUALITY_THRESHOLD = 0.7
    
    if state["retrieval_round"] >= MAX_ROUNDS:
        return "synthesize"  # 达到最大轮次，强制生成
    
    latest_score = state["eval_scores"][-1] if state["eval_scores"] else 0
    if latest_score >= QUALITY_THRESHOLD:
        return "synthesize"  # 质量达标，进入生成
    
    return "rewrite"  # 质量不足，重新改写查询

# --- 构建 Graph ---
workflow = StateGraph(AgenticRAGState)

# 添加节点
workflow.add_node("rewrite", query_rewriter)
workflow.add_node("retrieve", retriever)
workflow.add_node("evaluate", evaluator)
workflow.add_node("synthesize", answer_synthesizer)

# 添加边
workflow.set_entry_point("rewrite")
workflow.add_edge("rewrite", "retrieve")
workflow.add_edge("retrieve", "evaluate")

# 条件边：评估结果决定是重试还是生成
workflow.add_conditional_edges(
    "evaluate",
    should_retry,
    {
        "rewrite": "rewrite",
        "synthesize": "synthesize"
    }
)

workflow.add_edge("synthesize", END)

# 编译
app = workflow.compile()

# 执行
result = app.invoke({
    "user_query": "Kubernetes生产环境中如何设计高可用架构？",
    "retrieval_round": 0,
    "max_rounds": 3
})
print(result["final_answer"])

    rewrite --> retrieve[Retriever]
    retrieve --> evaluate[Evaluator]
    evaluate -->|score ≥ 0.7| synthesize[Answer Synthesizer]
    evaluate -->|score < 0.7| rewrite
    synthesize --> END((( )))

6. 评估指标

6.1 检索质量指标

指标 公式 / 含义 适用场景
Precision@K $\frac{\text{Top-K中相关文档数}}{K}$ 评估检索结果的精确度
Recall@K $\frac{\text{Top-K中相关文档数}}{\text{总相关文档数}}$ 评估检索结果的覆盖率
MRR（Mean Reciprocal Rank） $\frac{1}{N}\sum_{i=1}^{N}\frac{1}{\text{rank}_i}$ 评估第一个相关文档的排名位置
NDCG@K 考虑排序位置的归一化折损累积增益 评估排序质量，位置越靠前权重越大

6.2 生成质量指标

指标 含义 评估方式
Answer Relevance 答案与问题的语义相关性 LLM-as-Judge / 语义相似度
Faithfulness 答案是否忠实于检索文档（不编造） NLI 模型 / Factual Consistency Check
Context Precision 检索到的上下文中有多少被真正用到 答案与检索文档的 token overlap
Context Recall 答案所需信息是否都被检索覆盖 Ground truth 信息覆盖度

6.3 Agentic RAG 特有的端到端指标

  • Round Efficiency（轮次效率）：平均需要几轮检索才能获得满意答案
  • Rewrite Effectiveness（重写有效性）：查询重写后检索质量提升的幅度
  • Self-Correction Rate（自我纠错率）：Agent 自行发现并纠正错误的比例

7. 生产落地建议

7.1 向量库选择

向量库 特点 适用场景
Milvus 分布式、高性能、支持混合搜索 大规模生产环境（百万级以上向量）
Qdrant Rust 实现、高性能、过滤能力强 中大规模、有复杂过滤需求
Weaviate 内置 hybrid search、支持 GraphQL 需要开箱即用 hybrid search
Chroma 轻量、Python native、零配置 原型验证、小规模应用
Pinecone 全托管、免运维 不想自己维护基础设施的团队

7.2 Chunk 策略

推荐策略：Adaptive Chunking（自适应分块）

1. 基础分块：Token-based，512 tokens，overlap 64 tokens
2. 语义分块：基于句号/段落分隔，保持语义完整性
3. 层级分块（Hierarchical）：
   - Level 1：文档级 chunk（整篇文章）
   - Level 2：段落级 chunk（3-5段）
   - Level 3：句子级 chunk（1-3句）
   检索时先匹配小 chunk，再拉取上级 context
4. Small-to-Big 检索：用小 chunk 做检索，返回大 chunk 的 context

7.3 Hybrid Search + Reranker 管线

[Query] 
  → Dense Retrieval (向量相似度，Top-50)
  → Sparse Retrieval (BM25，Top-50)
  → Reciprocal Rank Fusion (融合 Top-100 → Top-20)
  → Cross-encoder Reranker (精排 Top-20 → Top-5)
  → [最终检索结果]

推荐 Reranker 模型：

  • BGE-Reranker-v2（BAAI）：中英文表现优异，开源
  • Cohere Rerank：API 服务，效果好但需付费
  • Jina Reranker：支持更长 context（8K tokens）

7.4 成本与延迟优化

优化策略 效果 代价
缓存（Cache） 相同 query 命中缓存直接返回 存储成本
Early Exit 评估通过 → 不触发额外检索 需调优阈值
Streaming 边生成边返回，降低体感延迟 对 streaming 有格式要求
异步执行 多子查询并行检索 代码复杂度上升
模型蒸馏 评估器和重写器用小模型 评估精度可能下降

7.5 监控与可观测性

生产环境需要监控以下关键指标：

  • 检索延迟 P50/P95/P99
  • 检索质量分布（每轮的 relevance score 分布）
  • 重试率（触发 Query Rewrite 的比例）
  • 用户满意度（点赞/点踩率、会话完成率）
  • 幻觉率（Faithfulness 评分低于阈值的比例）

核心总结：Agentic RAG 不是标准 RAG 的"升级版"，而是一次范式转变——从"检索-生成"的单向流水线，进化为"检索-评估-决策-再检索-生成"的智能回路。在这个回路中，Agent 的角色不是替代 RAG，而是赋予 RAG 自我感知和自我纠错的能力，使其在面对模糊查询、多跳推理、信息冲突等复杂场景时，不再是一个盲目的"文档搬运工"，而是一个有判断力的"信息侦探"。
