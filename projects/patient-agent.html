项目定位

患者侧智能辅助 Agent 原型系统，覆盖诊前、诊中、诊后全场景。不是单纯的对话机器人，而是一个集成了身份认证、多层记忆、结构化数据查询、知识检索、多模态理解和语音输出的智能体系统。当前版本是"可运行、可测试、可解释、可迭代"的原型，重点验证以下能力能否稳定协同：患者事实查询、会话连续记忆、长期摘要记忆、知识块检索、工具调用与问答编排。

一次问答的完整生命周期

用户输入（文本 / 图片）
    │
    ▼
Step 1: 身份解析层
    │  Auth Token 校验 / 自然语言身份提取 / Session ID 分配
    │  → 结果: patient_id + hospital_id
    │  HMAC-SHA256 签名的 token，含载荷 + 过期时间
    │  回退: 从对话文本提取姓名/出生年/手机号，与患者库模糊匹配
    │
    ▼
Step 2: 图片分析层（可选）
    │  调用视觉 LLM 提取图片摘要（报告文字/药品包装/检查结果/患处照片）
    │  自动降级：首选 Vision 模型失败 → 次选 → 通用模型
    │
    ▼
Step 3: 短期工作记忆更新
    │  最近 5 轮消息 + 会话状态（意图/主题/目标）+ 活跃实体 + 风险信号
    │
    ▼
Step 4: 长期记忆上下文构建
    │  患者主档 + 业务画像 + 对话画像 + 关键事件 + 偏好 + 病历/就诊
    │
    ▼
Step 5: 知识检索（RAG）
    │  混合检索：向量 + 关键词 + 元数据 + 时效 → 四维加权排序
    │
    ▼
Step 6: 意图识别 + 规划
    │  模板优先（快、可解释）→ LLM 兜底（广、全面）
    │  共识投票选最优执行路径
    │
    ▼
Step 7: 工具执行（MCP Server）
    │  签发身份 token / 校验身份 / 查病历 / 查就诊 / 查聚合画像
    │
    ▼
Step 8: 答案生成（LLM）
    │  组装 Prompt: 身份 + 短期记忆 + 长期记忆 + 知识 + 工具结果
    │
    ▼
Step 9: 后处理 & 输出
    │  Markdown 清理 → 语音合成 (TTS) → 记忆持久化

四层记忆架构

┌─────────────────────────────────────────────────────────┐
│  工作记忆（内存中，最近5轮，高频/小容量）                    │
│  recent_messages | intent | current_topic | goal          │
│  working_summary | next_action | memory_focus              │
│  active_entities（药物/症状/检查/指标）                     │
│  risk_signals（红旗/用药/监测）                             │
├─────────────────────────────────────────────────────────┤
│  长期摘要记忆（DB，持续更新摘要）                           │
│  MemoryUserProfile | MemoryBusinessProfile |              │
│  MemoryConversationProfile | MemoryPreference |           │
│  MemoryKeyEvent（关键事件，幂等合并，不重复）               │
├─────────────────────────────────────────────────────────┤
│  事实层（DB，最底层最可信的数据来源）                       │
│  患者主档 + 病历（最近3条）+ 就诊记录（最近3条）             │
│  属于"请求期临时拼装"，不是持久化事实向量层                  │
├─────────────────────────────────────────────────────────┤
│  知识记忆（ChromaDB + 混合 RAG）                          │
│  知识块元数据 + 向量检索 + 关键词补召回 + 综合排序           │
│  可设置 effective_from / expires_at 版本控制               │
└─────────────────────────────────────────────────────────┘

记忆系统设计了记忆金字塔缓存模式：越往上越热、越小、越临时；越往下越冷、越大、越持久。短期工作记忆采用结构化方式（而非简单文本窗口），包含 intent、current_topic、goal、working_summary、next_action 等字段。长期摘要记忆在每次对话结束后更新，支持幂等合并（相同 canonical_key 的事件合并而非重复）。

两种对话模式

模式适用场景行为差异
general普通问答不写入短期消息到 DB，只维护内存工作记忆
memory需要持久化上下文每次消息写入 memory_conversation_messages 表，跨会话回溯

混合 RAG 检索策略（四维加权）

final_score = 0.58 × vector_score     # 语义相似度（余弦）
            + 0.22 × keyword_score    # 字面匹配（Jaccard）
            + 0.12 × metadata_score   # 医院/领域匹配
            + 0.08 × recency_score    # 新鲜度（指数衰减）
            + 0.04 × confidence       # 置信度

嵌入层降级链：OpenAI Embeddings → HuggingFace (BAAI/bge-small-zh-v1.5) → HashTextEmbedder（确定性哈希，不依赖外部模型）。向量模型 BAAI/bge-small-zh-v1.5，768 维。

三层安全防线（患者过敏保护）

系统设计了五层防护防止过敏药物推荐等风险，分两层场景应对：

场景 A：已知过敏

患者档案记录"阿司匹林过敏"，RAG 知识库推荐头痛标准治疗含阿司匹林

  1. 过敏药物提取：从 allergy_history 字段解析药物名（基于预定义药物词典 + 正则回退）
  2. RAG 知识过滤：检索后逐行扫描，剔除含过敏药物的行
  3. Prompt 硬指令注入："PATIENT ALLERGY WARNING: 患者对 阿司匹林 过敏。绝不要推荐"
  4. 回答后安全扫描：LLM 生成后扫描回答文本，命中时追加醒目警告

场景 B：未知过敏

患者档案 allergy_history 为空，系统不知道患者是否过敏

  1. 检测到档案无过敏记录时设置 allergy_history_unknown = True
  2. Prompt 注入"全面回应"行为指令，在回答行文中自然包含用药提醒
  3. 互斥逻辑：已知过敏 → 硬警告；未知过敏 → 自然融入；无患者绑定 → 均不注入

双因子身份校验链

验证 token 签名（HMAC-SHA256）→ 验证未过期 → 查询患者存在且 active → 验证 hospital_id 一致 → 通过。token 中的 patient_id 与传入的一致（否则 403），hospital_id 与患者记录一致（否则 403）。

意图识别与规划

"模板优先 + LLM 兜底"分层策略：命中关键词模板（如"就诊"→ visit_records_query）直接返回（快、确定性高）；未命中则调用 LLM 分类（慢但覆盖广）。预定义意图包括：就诊记录查询、病历查询、症状咨询、患者画像总结、多模态摘要、通用医学问答。

计划生成 + 共识投票：每个意图映射到 1-2 个工具调用步骤。对复杂情况，LLM 可生成多候选计划，系统通过共识投票选择执行路径。

身份认证三种方式

方式来源优先级
Auth Token显式传入最高 — HMAC-SHA256 签名，含 patient_id + hospital_id + 过期时间
自然语言提取对话文本中 — 自动提取姓名、出生年、手机号，模糊匹配患者库
直接传参API 参数低 — 直接传 patient_id，适用于测试

多模态理解 + 语音输出

  • 图片理解：支持检查报告照片、药品包装、患处照片。Base64 编码后传给视觉 LLM，自动降级回退
  • 语音播报：双引擎支持 — OpenAI Compatible TTS（远程，音质高）和 Kokoro（本地，隐私好），网络失败自动降级

Prompt 组装策略

最终 Prompt 按顺序拼接：系统角色指令（含安全约束、偏好风格）→ 短期记忆（最近 5 轮 + 会话状态）→ 长期记忆（患者画像 + 关键事件 + 偏好）→ 知识上下文（RAG 检索）→ 工具执行结果 → 用户问题。

设计模式总结

  • 多层缓存/记忆金字塔：工作内存（热/小/临时）→ 短期消息（DB，有限时间）→ 关键事件（DB，长期）→ 画像/偏好（DB，几乎不变）
  • 规则优先 + LLM 兜底：在意图识别、实体提取、身份解析等多个环节先走确定性规则（快、零成本），失败时调用 LLM
  • 降级链：嵌入层、视觉层、语音层均有首选 → 次选 → 本地兜底方案
  • 匿名→实名过渡：匿名会话缓冲（不关联患者 ID）→ promote（身份确认后迁移）→ 正式消息

关键 API

  • 数据接口：POST/GET /api/v1/patients、病历、就诊记录 CRUD
  • Memory 接口：GET /api/v1/memory/profile、conversations/messages、knowledge-chunks/retrieve
  • Agent 接口：POST /api/v1/mcp/agent/query、query-with-image、speech
