企业多专家 Agent 系统

PythonReActChroma BM25DeepSeekFastAPI SSEJWTFSMDocker

项目定位

一个对话驱动的企业智能工作台，核心理念是：说话即可完成所有日常工作事务，无需学习菜单和表单。系统覆盖 HR 人事（请假/考勤/薪酬）、IT 支持（工单/设备/软件）、财务报销（差旅/预算/薪资）、法务合规（合同/制度/合规检查）四大业务领域。

交互设计原则

系统支持两种交互模式：

  • 纯对话：知识查询、制度咨询、状态查询（如"年假政策是什么""我的申请进度"）
  • 卡片交互：有固定字段的操作类意图，Agent 返回 action_card JSON，前端自动渲染表单（请假申请、报销申请、设备报修、打卡、审批等 7 种卡片类型）

卡片提交后直接调用对应 REST API，结果以 Agent 消息形式展示在对话中。当用户说"我要请假"时，系统识别意图后渲染 leave_form 卡片，预填充类型（年假），用户填写后一键提交。

Agents 层 — 三级路由引擎

确定性优先：规则 → 关键词 → LLM

层级触发条件confidence延迟
寒暄检测正则匹配"你好/谢谢/你是谁"1.0（固定）<1ms
关键词匹配87 关键词 × 3 领域，动态 confidence0.76~0.95<1ms
LLM 路由LLM Structured Output，temperature=0.0LLM 返回~500ms

动态 confidence 公式：单领域 = 0.75 + 0.20 × (命中数 / 领域总词数)，上限 0.95；多领域差距 ≥ 2x 时 = 0.70 + 0.10 × min(倍数-1, 1.5)，上限 0.85。

RAG 检索管线

  1. 查询改写：LLM 将口语转为书面语（如"年终奖什么时候发" → "年终奖发放规则 发放时间"），失败时静默回退
  2. 向量检索：ChromaDB + BAAI/bge-large-zh-v1.5（1024 维），余弦相似度
  3. BM25 检索：rank-bm25 + jieba 分词，权重为向量结果的 1.5x
  4. RRF 融合：score = 1/(k + rank_vector) + 1.5/(k + rank_bm25)，k=60
  5. 重排序（可选）：CrossEncoder (bge-reranker-base) 从 20 条候选中精选 3 条
  6. 缓存：MD5 key + 60s TTL，相同查询直接命中

Query 模式检索 top_k=3，Action 模式 top_k=10（需要更多上下文支持工具决策）。

Skills 层 — 复合业务能力（v2 新增）

在 Tools 原子操作之上抽象出的复合业务层，解决多步骤状态流转问题：

工单生命周期（状态机）

待处理 ─→ 处理中 ─→ 已完成
  │          │
  └── 已关闭 ←┘
       │
       └── 待处理 (重开)

严格 FSM 状态转换校验，非法转换返回 error。支持：创建工单（自动按类型路由到对应部门经理）、处理、完成、关闭、重开、升级（优先级变为"高"并通知部门经理）。Agent / CLI / 定时任务均可调用。

通用审批引擎

统一了请假、报销的审批流程，消除多个工具文件中重复的审批逻辑。核心思路：规则配置 + 汇报链 = 审批步骤。

  • 请假：≤3 天 → 直属上级；≤7 天 → 直属上级 + 部门负责人；>7 天 → + HR 总监
  • 报销：<5000 元 → 直属上级；≥5000 元 → 直属上级 + 财务经理
  • 驳回整条记录，自动推进下一级审批，完成时扣减假期余额

通知中心 + 聚合报表

统一通知发送（5 分钟同内容去重），批量标记已读。聚合报表覆盖：工单日报/周报、月度请假聚合、预算使用率（超 90% 预警）、综合仪表盘。

安全网关（v2 新增）

  • JWT 双 Token：Access Token + Refresh Token，登出立即撤销
  • 密码安全：pbkdf2_hmac 哈希（60 万轮迭代）
  • 输入净化：NFKC 正规化 + 控制字符剔除
  • 速率限制：登录 5 次失败锁定 15 分钟

跨域 Handoff 机制

两个触发条件：Orchestrator 识别多领域问题，或 Agent 主动返回 handoff_suggestions。交割上下文包含：来源 Agent、已处理完成项、待处理项、待执行操作、共享用户信息（user_id / department）。汇总时两个 Agent 的回答合并到一个 final_answer 中，标记来源。

可观测性体系

  • Trace 全链路追踪：routing → retrieval → tool_call → llm_call → answer，JSONL 落盘
  • 自动化评估：20 条测试集评估路由准确率（当前 95.0%），CLI 一键运行
  • 前端 Trace 查看器：在 Web UI 中查看每次对话的完整链路上报

对话操作卡片模板

系统支持 7 种操作卡片，前端自动渲染：

卡片类型触发意图提交接口
leave_form请假申请POST /api/leaves
expense_form报销申请POST /api/expenses
ticket_formIT 维修工单POST /api/tickets
attendance_punch考勤打卡POST /api/attendance/punch
overtime_form加班申请前端雏形
trip_form出差申请前端雏形
approval_action审批操作查看待审批列表

Agent 回复中携带 action_card JSON 字段，前端识别后在消息下方渲染对应组件。新增模板只需改动 4 个位置：数据库/工具层 → 后端 API → 意图识别规则 → 前端卡片渲染。

可配置参数

LLM_MODEL=deepseek-v4-flash
LLM_BASE_URL=https://api.deepseek.com
PLANNER_ENABLED=false          # 任务规划
MEMORY_ENABLED=false           # 会话记忆，窗口大小 5
RERANK_ENABLED=false           # 检索重排序
TOP_K_RETRIEVAL=10             # RAG 检索返回条数
RERANK_TOP_K=3                 # 重排序后精选条数
CHUNK_SIZE=500 / CHUNK_OVERLAP=80
RAG_CACHE_TTL_SECONDS=60       # 查询缓存 TTL

部署与测试

  • 启动：python -m src.main serve → 浏览器打开 http://localhost:8080
  • CLI：python -m src.main cli 交互式终端模式
  • 内网穿透：cloudflared / ngrok / cpolar 隧道暴露给演示对象
  • 测试脚本：test_routing.py（路由准确性）、test_rag.py（检索质量）、run_eval.py（20 条评估集）
  • Docker：docker compose up -d 一键部署
  • 工单 CLI：ticket_ops.py 创建/处理/完成/关闭/重开/升级/报表（11 个子命令）
  • 审批 CLI：approval_ops.py 待审批/审批通过/驳回/状态/提交（5 个子命令）

设计约束

  • 对话优先：所有操作都应能通过对话完成，表单/按钮是辅助加速
  • 最小权限：员工只能操作自己的数据，经理只能看下属，HR 全局可见
  • 审批不可绕过：所有涉及资源的申请必须经过审批流
  • 操作可追溯：所有工具调用写入 audit_log，支持事后审计
