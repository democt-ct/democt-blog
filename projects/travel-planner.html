AI 旅行规划师

PythonFastAPI高德地图 Cloudflare WorkersPOI Pipeline

项目定位

对话驱动的智能旅行规划系统。用户通过自然语言描述需求（时间、地点、兴趣偏好），系统自动生成结构化行程，并将每个行程点精准映射到高德地图上展示。核心理念：LLM 只负责语义理解与候选生成，不直接作为坐标来源——只有通过高德验证的地点才能进入路线规划。

POI 解析 Pipeline（完整链路）

用户: "想去南头古城、华侨城和海岸城逛逛"
    │
    ▼
Step 1: LLM 地点需求解析
    │  从自然语言中识别地点表达，区分精确地名 vs 泛地点
    │  "南头古城"→ 精确, "逛逛"→ 泛地点
    │
    ▼
Step 2: 候选地点生成
    │  LLM 为每个需求生成多个候选：完整名称 + 变体 + 区域锚点
    │  泛地点(如"美食街") → 短关键词变体（"美食街 深圳"）
    │
    ▼
Step 3: 高德 POI 搜索验证
    │  唯一真值来源！调用高德搜索/地理编码 API
    │  精确匹配 → 直接落图
    │  模糊匹配 → 锚点兜底（城市中心坐标）
    │
    ▼
Step 4: 地点清洗
    │  过滤误识别值（如品牌名被当作地点）
    │  清洗语义尾缀（"深圳南头古城"→ "南头古城"）
    │  泛地点自动短关键词变体回退
    │
    ▼
Step 5: 验证通过 → 进入路线规划
    │  失败 → 降级处理，不硬编坐标

分层的 POI 处理

  • 需求层：LLM 理解用户语义，从自然语言中提取地点需求
  • 候选层：LLM 针对每个需求生成多个候选地点（含完整名称 + 变体）
  • 搜索层：高德 POI 搜索 / 地理编码（唯一真值来源）
  • 验证层：精确匹配 → 直接落图；模糊匹配 → 兜底
  • 最终过滤：合并去重，跨天重复检测，地点类型/角色分离

地点类型与角色分离

每个地点独立分配「地点类型」和「行程角色」，精确过滤标准：只有当候选的「类型 + 关键词」与用户需求一致时才进入排序，不在候选列表则从高德搜索排名最高的 POI。

Cloudflare Worker 架构

系统采用4 个独立 LLM 管道，基于 Cloudflare Workers + AI Gateway 部署：

  1. 需求解析管道：用户指令 → 结构化需求（时间/地点/偏好），含重试 + JSON Schema 验证
  2. 行程生成管道：需求 + 已验证 POI + 路线 → 结构化 iternery JSON
  3. 候选生成管道：需求 → 地点候选项（含变体）
  4. 地点解析管道：地点名 → 精确坐标落图

Workers 之间通过环境变量共享 AI Gateway 模型配置，统一使用 @cf/meta/meta-lama-3.1-70b-instruct（暂定）。管道的重试机制基于异常控制，JSON 解析失败自动重试（最多 3 次）。

会话管理

  • 创建会话：用户输入 → LLM 生成 iternery → 构造会话对象 → 存入 SQLite → 返回 session_id
  • 恢复会话：加载行程 JSON → 初始化地图 → 恢复 map/poi/route 状态
  • 删除会话：硬删除（delete）+ 软删除（archive），支持批量清理
  • 多会话切换：切换时同步最新行程到地图视图

前端交互细节

  • 行程自动落图：生成/编辑行程后自动同步到高德地图
  • 起点定位：每天第 1 个行程点作为路线规划起点
  • 2D/3D 切换：地图模式一键切换
  • 泛地点补点：备选点直接加入指定日期（不经 LLM 重规划）
  • 搜索与路线共存：手动地点搜索与当前已有行程并行显示
  • 城市清洗兜底：坐标解析失败时自动定位城市中心
  • 关键词兜底：空结果时自动用短关键词变体重试

留存边与待优化点

  • 高德配额限制：地理编码/搜索 API 有每日调用上限
  • 定位权限：首次同步仍有多次请求（已用缓存减轻）
  • 行程文本不自动重写：编辑后需要额外触发 LLM 重描述
  • 泛地点补点使用高频候选：尚未整合用户历史偏好做个性化推荐
