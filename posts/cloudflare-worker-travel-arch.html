AI 旅行规划师的 Cloudflare Worker 架构

2026-04-25Cloudflare架构旅行

AI 旅行规划师项目不同于传统后端单体架构，选择了 Cloudflare Workers 作为运行时环境。

整体架构

用户浏览器
  │
  ├─ GET / → 静态资源 (index.html + JS + CSS)
  ├─ POST /chat → 对话入口（收集需求 / 生成行程 / 重规划）
  ├─ POST /map/search → 高德 POI 搜索
  ├─ POST /api/map/route/plan → 高德路径规划
  └─ GET /sessions → 会话管理
  │
  ▼
Cloudflare Worker (src/index.ts) ← 单一入口，路由分发
  │
  ├─ LLM 调用 (OpenAI 兼容 API)
  │    ├─ 需求解析 (requirementInterpreter)
  │    ├─ 候选池构建 (candidatePoolBuilder)
  │    ├─ 城市签名种子 (citySignatureSeeder)
  │    └─ 候选审查 (candidatePlanner)
  │
  ├─ 高德 API
  │    ├─ POI 搜索 (text search)
  │    ├─ 路线规划 (walking / driving)
  │    ├─ 逆地理编码 (坐标→城市)
  │    └─ 天气查询
  │
  ├─ 规划引擎 (travelPlanningPipeline)
  │    ├─ 候选池构建 → 评估 → 精选 → 输出
  │    └─ 并行与串行结合的 LLM 管道
  │
  └─ 会话管理 (SQLite/WAL + Durable Objects)

关键技术决策

  • Edge 运行时：Cloudflare Workers 全球部署，低延迟
  • LLM 管道：4 个专业 LLM 调用各司其职，不是单一 prompt 解决一切
  • 高德为唯一真值：所有地点坐标必须经过高德验证，LLM 不产生坐标
  • Durable Objects：会话状态持久化，支持断点续聊
